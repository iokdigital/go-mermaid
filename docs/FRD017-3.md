## 1. Overview

### 1.1 Summary

Build a standalone, domain-agnostic, pure Go library (`github.com/iokdigital/go-mermaid`) for generating and rendering Mermaid diagrams. The library covers the Mermaid diagram types relevant to workflow conversion tooling, produces output in multiple formats (`.mmd`, `.svg`, `.png`, `.html`, `.pdf`, `.json`, `.dot`), and exposes both file and streaming I/O. It is imported by `alteryx2talend` (and other tools in the monorepo) to replace the dependency on Node.js / Chromium / mermaid-cli.

### 1.2 Motivation

- Worker container is ~400вЂ“500 MB vs ~150 MB target; Node.js (~50 MB) + Chromium (~150 MB) + mermaid-cli (~50 MB) account for ~250 MB
- Existing `MermaidGenerator` is tightly coupled to `alteryx2talend` domain types, cannot be shared across tools, and emits deprecated `graph TD` syntax (Mermaid v1)
- No interface abstraction makes diagram generation untestable and unmockable
- mermaid-cli cannot run in restricted or air-gapped container environments
- Only 2 of ~15 Mermaid diagram types are currently used
- PNG output requires Node.js + Chromium today; a pure Go path removes this constraint

### 1.3 Goals

| Goal | Description |
|------|-------------|
| Standalone library | Domain-agnostic Go module; no knowledge of Alteryx or Talend types |
| Eliminate external deps | No Node.js, Chromium, npm, or mermaid-cli in any container |
| Expand diagram vocabulary | All diagram types relevant to workflow conversion tooling (see В§2) |
| Multi-format output | `.mmd`, `.svg`, `.png`, `.html`, `.pdf`, `.json`, `.dot` |
| Streaming I/O | All output targets `io.Writer`; file helpers are convenience wrappers |
| Go version agnostic | Minimum Go 1.20; no CGo; no platform-specific build tags |
| Reusable across tools | `talend2airflow`, `alteryx2airflow`, and future tools import the same library |

### 1.4 Library vs. Consuming Project Boundary

The `go-mermaid` library is **domain-agnostic**. It knows about Mermaid diagram types (flowchart, sequence, state, etc.) but knows nothing about `AlteryxWorkflow`, `AnalysisResult`, or any other domain type.

The **domain-to-diagram conversion layer** lives in each consuming project:

```

    в”њв”Ђв”Ђ ast/flowchart.go                 (AST)
    в”њв”Ђв”Ђ svg/renderer.go                    (render)
    в””в”Ђв”Ђ png/renderer.go                    (render)

github.com/iokdigital/alteryx2talend
    в””в”Ђв”Ђ internal/diagram/
        в””в”Ђв”Ђ gen/
            в”њв”Ђв”Ђ flowchart_gen.go           в†ђ AlteryxWorkflow в†’ FlowchartDiagram
            в”њв”Ђв”Ђ state_gen.go               в†ђ JobStatus в†’ StateDiagram
            в””в”Ђв”Ђ ...                        (imports go-mermaid types)
```

### 1.5 Scope

**In Scope (go-mermaid library):**
- Mermaid AST types for all Phase 1 diagram types
- `.mmd` encoder for all Phase 1 diagram types
- `.dot` encoder (Graphviz; migrated from existing `DOTGenerator`)
- Pure Go SVG renderer for flowchart diagrams (Phase 1)
- PNG output via SVG rasterization (pure Go, no CGo) with named resolution presets
- HTML output (online CDN mode only; see В§8.1)
- PDF output for flowcharts (via `go-pdf/fpdf`)
- JSON serialisation of the diagram AST
- Streaming `io.Writer` interface with file and `[]byte` convenience wrappers

**In Scope (alteryx2talend integration):**
- `internal/diagram/gen/` вЂ” domain-to-AST conversion functions
- `internal/diagram/generator.go` вЂ” updated to use new library
- `cmd/analyze.go` вЂ” new CLI flags
- `Dockerfile.worker` audit and Node.js/Chromium removal

**Out of Scope:**
- Support for every Mermaid diagram type (see В§2.3)
- Interactive / clickable diagram features (beyond optional node hyperlinks in SVG)
- Pixel-perfect match to official mermaid-cli SVG output
- Real-time / live-reload updates
- Mermaid.js API compatibility layer

### 2.1 Phase 1 вЂ” Core (must have)

| Type | Mermaid Keyword | Primary Use Cases |
|------|----------------|-------------------|

| Sequence Diagram | `sequenceDiagram` | Conversion pipeline steps, workerв†”API interactions, Celery task lifecycle |
| State Diagram | `stateDiagram-v2` | Job status lifecycle (pending в†’ queued в†’ running в†’ complete/failed/cancelled) |
| ER Diagram | `erDiagram` | Schema field mapping (input fields в†’ output fields) |
| Class Diagram | `classDiagram` | Component hierarchy, tool category taxonomy |

### 2.2 Phase 2 вЂ” Extended

| Type | Mermaid Keyword | Primary Use Cases |
|------|----------------|------------------|
| Pie Chart | `pie` | Conversion coverage statistics, confidence score distribution |
| Quadrant Chart | `quadrantChart` | Confidence vs. complexity analysis |
| Gantt | `gantt` | Migration project timeline |
| Mind Map | `mindmap` | Component taxonomy overview |

### 2.3 Explicitly Out of Scope

| Type | Reason |
|------|--------|
| `gitGraph` | No version control visualisation use case |
| `xychart-beta` | Pie/quadrant are more appropriate for this domain |
| `timeline` | Gantt covers the requirement |
| `block-beta` | Flowchart covers this |
| `packet` | Not relevant to workflow conversion |
| `architecture-beta` | Unstable Mermaid API; revisit when stable |

### 2.4 Existing Coverage Gap

The current `MermaidGenerator` in `alteryx2talend` emits the deprecated `graph TD` / `graph LR` syntax (Mermaid v1). The new library emits the modern `flowchart` keyword. The consuming project's generator must be updated to produce structurally identical output via the new AST. The keyword change (`graph LR` в†’ `flowchart LR`) is backward-compatible in all mermaid.js renders but will break any scripts or tests that do string-matching on raw `.mmd` output (see В§10.3 вЂ” Negative Tests).

## 3. Output Format Requirements

### 3.1 Supported Output Formats

| Format | Extension | Description | Renderer |
|--------|-----------|-------------|----------|
| Mermaid source | `.mmd` | Raw Mermaid text syntax | String builder |
| SVG | `.svg` | Scalable vector graphics | Pure Go renderer (В§6) |
| PNG | `.png` | Raster image at configurable resolution | SVG в†’ PNG rasterizer (В§7) |
| HTML | `.html` | Standalone HTML + mermaid.js (browser-rendered) | Template + CDN/embed (В§8) |
| PDF | `.pdf` | Portable document for reports | SVG в†’ PDF via `go-pdf/fpdf` (В§8) |
| JSON | `.json` | Diagram AST for programmatic use | `encoding/json` marshal |
| Graphviz | `.dot` | Graphviz DOT format | String builder (migrated) |
| Markdown | `.md` | Fenced mermaid code block for docs/Confluence | String wrapper |

### 3.2 Format Support Matrix by Diagram Type

PNG is the **preferred** visual output. HTML is the fallback when:
- The SVG renderer for that diagram type is not yet implemented (Phase 1 for non-flowchart types), **or**
- The estimated PNG output size would exceed the configured `MaxPNGBytes` limit

| Type | mmd | svg | png | html (fallback) | pdf | json | dot |
|------|:---:|:---:|:---:|:---------------:|:---:|:----:|:---:|
| Flowchart | вњ… | вњ… | вњ… | вњ… | вњ… | вњ… | вњ… |
| Sequence | вњ… | Phase 2 | Phase 2 | вњ… | Phase 2 | вњ… | вЂ” |
| State | вњ… | Phase 2 | Phase 2 | вњ… | Phase 2 | вњ… | вЂ” |
| ER | вњ… | Phase 2 | Phase 2 | вњ… | Phase 2 | вњ… | вЂ” |
| Class | вњ… | Phase 2 | Phase 2 | вњ… | Phase 2 | вњ… | вЂ” |

When a caller requests PNG for a type/phase combination where the PNG renderer is not available, the library returns `ErrRendererNotAvailable` with a `FallbackFormat` field set to `FormatHTML`, allowing the caller to retry with the fallback.

### 3.3 PNG Size Limit and HTML Fallback

Large workflows can produce very large PNG files. The library enforces a configurable maximum:

```go
type RenderOptions struct {
    // ...
    MaxPNGBytes int64 // default 10_000_000 (10 MB); 0 = no limit
}
```

Before rasterizing to PNG, the renderer estimates output size from the SVG viewport dimensions and scale factor. If the estimate exceeds `MaxPNGBytes`, it returns `ErrPNGSizeLimitExceeded` (which implements `FallbackFormat() OutputFormat` returning `FormatHTML`). Callers should handle this explicitly:

```go
data, err := renderer.RenderBytes(d, diagram.FormatPNG)
if errors.Is(err, diagram.ErrPNGSizeLimitExceeded) {
    data, err = renderer.RenderBytes(d, diagram.FormatHTML)
}
```

### 3.4 I/O Interface

```go
// Renderer is the primary output interface.
type Renderer interface {
    // RenderTo writes to any io.Writer (files, buffers, HTTP responses, pipes).
    RenderTo(w io.Writer, d Diagram, format OutputFormat) error

    // RenderBytes returns output as []byte (convenience wrapper over RenderTo).
    RenderBytes(d Diagram, format OutputFormat) ([]byte, error)

    // RenderToFile creates parent directories if needed, then writes output.
    RenderToFile(path string, d Diagram, format OutputFormat) error

    // ContentType returns the MIME type for a given output format.
    ContentType(format OutputFormat) string

    // RenderMarkdown wraps mmd source in a fenced code block.
    RenderMarkdown(d Diagram) string
}
```

`net/http` is intentionally **not** imported by the core `Renderer` interface. HTTP serving is provided as a standalone helper to avoid forcing a `net/http` dependency on callers that don't serve diagrams over HTTP:

```go
// ServeHTTP writes diagram output to an http.ResponseWriter with correct
// Content-Type and Content-Disposition headers. It is a standalone function,
// not part of the Renderer interface, so callers without net/http are unaffected.
func ServeHTTP(r Renderer, w http.ResponseWriter, d Diagram, format OutputFormat) error
```

### 3.5 Additional Output Notes

**HTTP response** вЂ” `http.ResponseWriter` satisfies `io.Writer`, so `RenderTo` works directly. Use the standalone `ServeHTTP` helper (see В§3.4) when you also need `Content-Type` and `Content-Disposition: attachment; filename="diagram.{ext}"` set automatically.

**Markdown embed** вЂ” `RenderMarkdown(d Diagram) string` wraps mmd in ` ```mermaid ... ``` ` fences. Useful for auto-generated `.md` files and the `flowport2jira` Confluence publishing workflow.

### 4.3 Core Interfaces and Types

```go
// diagram.go

package diagram

import (
    "errors"
    "io"
)

// Diagram is the base interface for all diagram types.
type Diagram interface {
    Type()  DiagramType
    Title() string
}

// DiagramType identifies the Mermaid diagram variant.
type DiagramType string

const (
    TypeFlowchart DiagramType = "flowchart"
    TypeSequence  DiagramType = "sequence"
    TypeState     DiagramType = "state"
    TypeER        DiagramType = "er"
    TypeClass     DiagramType = "class"
    TypePie       DiagramType = "pie"
    TypeQuadrant  DiagramType = "quadrant"
    TypeGantt     DiagramType = "gantt"
    TypeMindMap   DiagramType = "mindmap"
)

// OutputFormat identifies the rendering output format.
type OutputFormat string

const (
    FormatMMD      OutputFormat = "mmd"
    FormatSVG      OutputFormat = "svg"
    FormatPNG      OutputFormat = "png"
    FormatHTML     OutputFormat = "html"
    FormatPDF      OutputFormat = "pdf"
    FormatJSON     OutputFormat = "json"
    FormatDOT      OutputFormat = "dot"
    FormatMarkdown OutputFormat = "md"
)

// Sentinel errors.
var (
    // ErrInvalidFormat is returned when an unrecognised OutputFormat string is passed.
    ErrInvalidFormat = errors.New("unrecognised output format")

    // ErrRendererNotAvailable is returned when a valid format/type combination
    // is not yet implemented. Check FallbackFormat() for an alternative.
    ErrRendererNotAvailable = errors.New("renderer not available for this diagram type and format")

    // ErrPNGSizeLimitExceeded is returned when the estimated PNG output
    // would exceed RenderOptions.MaxPNGBytes. Check FallbackFormat() в†’ FormatHTML.
    ErrPNGSizeLimitExceeded = errors.New("estimated PNG size exceeds configured limit")

    // ErrDuplicateNodeID is returned at build time when two nodes share the same ID.
    ErrDuplicateNodeID = errors.New("duplicate node ID")
)

// FallbackFormatError wraps a sentinel error with a suggested fallback format.
type FallbackFormatError struct {
    Err      error
    Fallback OutputFormat
}

func (e *FallbackFormatError) Error() string        { return e.Err.Error() }
func (e *FallbackFormatError) Unwrap() error        { return e.Err }
func (e *FallbackFormatError) FallbackFormat() OutputFormat { return e.Fallback }
```

### 4.4 Render Options

```go
// options.go

package diagram

// RenderOptions configures the renderer. Use NewRenderOptions() for defaults.
type RenderOptions struct {
    // PNG resolution preset. Overrides Scale and DPI if set.
    // Values: ResolutionWeb, ResolutionScreen, ResolutionScreenHD, ResolutionPrint, ResolutionPrintHQ.
    Resolution Resolution

    // PNG scale factor (overridden by Resolution if set).
    // 1.0 = 96 DPI base, 2.0 = 192 DPI, ~3.125 = 300 DPI.
    Scale float64

    // DPI stored in PNG metadata (informational; does not affect pixel dimensions).
    DPI int

    // MaxPNGBytes is the estimated PNG size limit in bytes.
    // If exceeded, RenderTo returns ErrPNGSizeLimitExceeded.
    // Default: 10_000_000 (10 MB). Set to 0 to disable the limit.
    MaxPNGBytes int64

    // SVGPadding adds whitespace around the diagram (px). Default: 40.
    SVGPadding int

    // SVGMaxWidth / SVGMaxHeight cap the computed viewport. Defaults: 8000 Г— 6000.
    SVGMaxWidth  int
    SVGMaxHeight int

    // LayoutOptions fine-tunes the Sugiyama layout algorithm.
    Layout LayoutOptions
}

// Resolution is a named PNG output quality preset.
type Resolution string

const (
    // ResolutionWeb: 1Г— scale (~96 DPI) вЂ” standard browser/screen display.
    ResolutionWeb Resolution = "web"

    // ResolutionScreen: 2Г— scale (~192 DPI) вЂ” standard retina/HiDPI display (default).
    ResolutionScreen Resolution = "screen"

    // ResolutionScreenHD: 3Г— scale (~288 DPI) вЂ” high-density display or large monitor export.
    ResolutionScreenHD Resolution = "screen-hd"

    // ResolutionPrint: ~3.125Г— scale (300 DPI) вЂ” standard print quality.
    ResolutionPrint Resolution = "print"

    // ResolutionPrintHQ: ~4.167Г— scale (400 DPI) вЂ” high-quality print / press.
    ResolutionPrintHQ Resolution = "print-hq"
)

// ResolutionScale returns the rasterization scale factor for a given Resolution preset.
// Switch-based (not a map) so the compiler catches missing cases on exhaustiveness checks.
func ResolutionScale(r Resolution) float64 {
    switch r {
    case ResolutionWeb:      return 1.0
    case ResolutionScreen:   return 2.0
    case ResolutionScreenHD: return 3.0
    case ResolutionPrint:    return 3.125
    case ResolutionPrintHQ:  return 4.167
    default:                 return 2.0 // screen default
    }
}

// ResolutionDPI returns the DPI metadata value for a given Resolution preset.
func ResolutionDPI(r Resolution) int {
    switch r {
    case ResolutionWeb:      return 96
    case ResolutionScreen:   return 192
    case ResolutionScreenHD: return 288
    case ResolutionPrint:    return 300
    case ResolutionPrintHQ:  return 400
    default:                 return 192
    }
}

func NewRenderOptions() RenderOptions {
    return RenderOptions{
        Resolution:   ResolutionScreen,
        MaxPNGBytes:  10_000_000,
        SVGPadding:   40,
        SVGMaxWidth:  8000,
        SVGMaxHeight: 6000,
        Layout:       DefaultLayoutOptions(),
    }
}
```

### 5.1 Minimum Version: Go 1.20

| Feature used | Introduced |
|-------------|-----------|
| `go:embed` | Go 1.16 |
| `io.ReadAll` | Go 1.16 |
| `any` alias | Go 1.18 |
| `errors.Join` | Go 1.20 |

Minimum is **Go 1.20**. No generics are used in the public API (see В§5.2).

### 5.2 Generics Decision

Generics are **not used** in the public API. Typed constructor functions per package give the same compile-time safety without the complexity of generic interfaces at call sites. Example:

```go
// ast/flowchart.go вЂ” typed constructors instead of Generator[T]

// NewFlowchart creates an empty flowchart; further nodes/edges added via builder methods.
func NewFlowchart(title string, direction Direction) *FlowchartDiagram

// Builder methods return the receiver for chaining.
// AddNode returns ErrDuplicateNodeID (at call time, not render time) if a node with
// the same ID has already been added.
func (f *FlowchartDiagram) AddNode(n *FlowNode) (*FlowchartDiagram, error)
func (f *FlowchartDiagram) AddEdge(e *FlowEdge) *FlowchartDiagram
func (f *FlowchartDiagram) AddSubgraph(s *Subgraph) *FlowchartDiagram

// Accessors (read-only views for renderers; not exported slices).
func (f *FlowchartDiagram) Type()      diagram.DiagramType
func (f *FlowchartDiagram) Title()     string
func (f *FlowchartDiagram) Direction() Direction
func (f *FlowchartDiagram) Nodes()     []*FlowNode
func (f *FlowchartDiagram) Edges()     []*FlowEdge
func (f *FlowchartDiagram) Subgraphs() []*Subgraph
```

Revisit generics if a strongly-typed `Generator[T]` pattern is needed in Phase 2.

### 5.3 Compatibility Guidelines

| Rule | Detail |
|------|--------|
| CGo prohibited | All dependencies must be pure Go |
| No `exec.Command` | No shelling out to external processes |

## 6. SVG Renderer Technical Specification

### 6.1 Scope (Phase 1: Flowchart Only)

The Phase 1 SVG renderer targets `FlowchartDiagram` exclusively. Non-flowchart types requesting SVG or PNG return `*FallbackFormatError{Err: ErrRendererNotAvailable, Fallback: FormatHTML}`.

### 6.2 Rendering Pipeline

```
FlowchartDiagram (AST)
        в”‚
        в–ј
   Layout Engine
   в”њв”Ђв”Ђ 1. Cycle detection      DFS; mark back-edges; reverse for layout
   в”њв”Ђв”Ђ 2. Layer assignment     Longest-path algorithm (O(V+E))
   в”њв”Ђв”Ђ 3. Crossing reduction   Barycenter heuristic, 2 passes (forward + backward)
   в””в”Ђв”Ђ 4. Coordinate assignment Median placement with minimum separation
        в”‚
        в–ј
   SVG Builder
   в”њв”Ђв”Ђ <defs>   Arrowhead markers, embedded CSS classes (no external stylesheets)
   в”њв”Ђв”Ђ Subgraph boxes          Behind node layer
   в”њв”Ђв”Ђ Edges                   Polyline with waypoints at rank boundaries
   в”њв”Ђв”Ђ Nodes                   Shape-appropriate SVG elements per NodeShape
   в”њв”Ђв”Ђ Labels                  <text> with <tspan> for wrapping; ellipsis truncation
   в””в”Ђв”Ђ Hyperlinks              <a href="{URL}"> wrapping node group when URL is set
        в”‚
        в–ј
   []byte  (valid SVG 1.1, self-contained, no external refs, flat fills only)
```

### 6.3 Layout Options

```go
type LayoutOptions struct {
    NodeSpacingH int // horizontal gap between nodes (px); default 60
    NodeSpacingV int // vertical gap between nodes (px); default 40
    RankSpacing  int // spacing between rank layers (px); default 80
}

func DefaultLayoutOptions() LayoutOptions {
    return LayoutOptions{NodeSpacingH: 60, NodeSpacingV: 40, RankSpacing: 80}
}
```

### 6.4 Layout Algorithm Details

| Step | Algorithm | Complexity | Notes |
|------|-----------|-----------|-------|
| Cycle detection | DFS back-edge marking | O(V+E) | Reversed edges drawn dashed after layout |
| Layer assignment | Longest-path | O(V+E) | Assigns each node a rank |
| Crossing reduction | Barycenter heuristic, 2 sweeps | O(VВ·E) | Adequate for <200 nodes |
| Coordinate assignment | Median placement | O(V) | Full Brandes-KГ¶pf deferred to Phase 2 |
| Edge routing | Polyline with rank-boundary waypoints | O(E) | Full orthogonal routing deferred to Phase 2 |

### 6.5 Viewport Computation

Viewport is **computed** from layout output вЂ” never hardcoded.

```
viewBox = "0 0 {W} {H}"
W = clamp(layout_width  + 2 Г— SVGPadding, 400, SVGMaxWidth)
H = clamp(layout_height + 2 Г— SVGPadding, 300, SVGMaxHeight)
```

If computed dimensions would exceed `SVGMaxWidth Г— SVGMaxHeight`, the renderer logs a warning and suggests splitting the diagram. The SVG is still generated at the capped size.

### 6.6 Text Handling

| Label length | Behaviour |
|-------------|-----------|
| в‰¤ 20 chars | Single `<text>` line |
| 21вЂ“40 chars | Word-wrapped via `<tspan>` elements |
| > 40 chars | Truncated to 37 chars + `вЂ¦` |

### 6.7 Self-loops and Cycles

- **Self-loops**: rendered as a small arc returning to the same node with a `вџі` glyph on the edge label
- **Back edges** (cycle members): drawn as dashed arcs bypassing the rank stack; do not participate in crossing reduction

### 6.8 Confidence Color Scheme

Matches existing `MermaidGenerator` color constants (backward compatibility with existing tests):

| Confidence | Fill | Stroke |
|-----------|------|--------|
| в‰Ґ 0.90 | `#90EE90` (light green) | `#2d862d` |
| 0.70вЂ“0.89 | `#FFD700` (gold) | `#b38f00` |
| < 0.70 | `#FFB6C1` (light pink) | `#c0392b` |
| 0.0 (not set) | `#f8fafc` (neutral) | `#64748b` |

### 6.9 SVG Constraints for oksvg Compatibility

The SVG renderer must stay within the subset that `oksvg` can rasterize (required for PNG). Prohibited:
- CSS filters (`filter:`, `blur()`, `drop-shadow()`)
- Gradient fills (`linearGradient`, `radialGradient`)
- Pattern fills
- `<use>` elements referencing external documents
- `clip-path` with complex paths

Permitted: flat fills, strokes, `<rect>`, `<circle>`, `<polygon>`, `<path>` (cubic BГ©zier and lines), `<text>`, `<tspan>`, `<a>`, `<defs>` with marker elements.

### 6.10 Optional Node Hyperlinks

When `FlowNode.URL` is non-empty, the SVG renderer wraps the node's group element:

```xml
<a href="{URL}" target="_blank" rel="noopener noreferrer">
  <!-- node shapes and label -->
</a>
```

This is useful for linking diagram nodes to FlowPort job detail pages or Confluence sections.

## 7. PNG Renderer

PNG is produced by rasterizing the flowchart SVG output using `oksvg` + `rasterx`.

### 7.1 Resolution Presets

| Preset | Scale | DPI metadata | Typical use |
|--------|-------|-------------|-------------|
| `web` | 1.0Г— | 96 DPI | Browser/wiki embed |
| `screen` | 2.0Г— | 192 DPI | Retina display export (default) |
| `screen-hd` | 3.0Г— | 288 DPI | High-density display, large monitor |
| `print` | 3.125Г— | 300 DPI | Standard print quality |
| `print-hq` | 4.167Г— | 400 DPI | High-quality print / press |

The preset is selected via `RenderOptions.Resolution`. Custom scale/DPI values can be set directly on `RenderOptions.Scale` / `RenderOptions.DPI` (preset takes precedence if both are set).

### 7.2 Implementation

```go
// png/renderer.go

func RenderToPNG(svgData []byte, w io.Writer, opts diagram.RenderOptions) error {
    scale := resolveScale(opts)

    icon, err := oksvg.ReadIconStream(bytes.NewReader(svgData), oksvg.WarnErrorMode)
    if err != nil {
        return fmt.Errorf("parse svg for png: %w", err)
    }

    width  := int(math.Ceil(icon.ViewBox.W * scale))
    height := int(math.Ceil(icon.ViewBox.H * scale))

    // Enforce size limit
    if opts.MaxPNGBytes > 0 {
        estimated := int64(width * height * 4) // RGBA bytes, pre-compression
        if estimated > opts.MaxPNGBytes {
            return &diagram.FallbackFormatError{
                Err:      diagram.ErrPNGSizeLimitExceeded,
                Fallback: diagram.FormatHTML,
            }
        }
    }

    img := image.NewRGBA(image.Rect(0, 0, width, height))
    draw.Draw(img, img.Bounds(), image.White, image.Point{}, draw.Src)

    scanner := rasterx.NewScannerGV(width, height, img, img.Bounds())
    rast    := rasterx.NewDasher(width, height, scanner)
    icon.SetTarget(0, 0, float64(width), float64(height))
    icon.Draw(rast, 1.0)

    return png.Encode(w, img)
}
```

## 8. HTML and PDF Output

### 8.1 HTML Renderer

Produces a self-contained single HTML file using an online CDN script tag. The primary use case is browser-based viewing of diagram types that do not yet have a Go SVG renderer (all non-flowchart types in Phase 1).

```html
<!DOCTYPE html>
<html lang="en">
<head>
  <meta charset="UTF-8">
  <meta name="viewport" content="width=device-width, initial-scale=1.0">
  <title>{{.Title}}</title>
  <script src="https://cdn.jsdelivr.net/npm/mermaid@11/dist/mermaid.min.js"></script>
</head>
<body>
  <pre class="mermaid">
{{.MermaidSource}}
  </pre>
  <script>mermaid.initialize({startOnLoad:true, theme:'default'});</script>
</body>
</html>
```

**Scope:** Online CDN mode only. No `go:embed`, no bundled mermaid.js, no offline support. The library has no knowledge of mermaid.js versions. If air-gapped HTML output is needed in a future release, the consumer can substitute the CDN URL with a locally hosted script вЂ” the HTML template will be designed to make this substitution straightforward via a `CDNOverrideURL` field in `RenderOptions`.

### 8.2 PDF Renderer

Uses `github.com/go-pdf/fpdf` to embed the rasterized PNG into a PDF page. **Limitation:** the pipeline is SVG в†’ PNG в†’ PDF (raster embed), not vector. The PDF will pixelate when zoomed beyond the render resolution. This is a known constraint of using `go-pdf/fpdf` without a vector SVG-to-PDF path; document this in the public API and README.

```go
// pdf/renderer.go

func RenderToPDF(svgData []byte, w io.Writer, title string, opts diagram.RenderOptions) error {
    // 1. Rasterize SVG в†’ PNG in memory at print resolution
    printOpts := opts
    printOpts.Resolution = diagram.ResolutionPrint
    var pngBuf bytes.Buffer
    if err := png.RenderToPNG(svgData, &pngBuf, printOpts); err != nil {
        return fmt.Errorf("rasterize for pdf: %w", err)
    }

    // 2. Embed PNG in A4 landscape PDF
    pdf := fpdf.New("L", "mm", "A4", "")
    pdf.AddPage()
    pdf.SetTitle(title, false)
    // scale PNG to fit page with margins; maintain aspect ratio
    // ...
    return pdf.Output(w)
}
```

### 9.5 Dockerfile.worker

The `go-mermaid` library has no Docker image вЂ” it is a pure Go module compiled into the `alteryx2talend` binary. The Docker impact is on `alteryx2talend`'s worker container only.

**Audit result (completed 2026-06-03):**

A full search of `Dockerfile.worker`, `worker/tasks/*.py`, and `worker/*.py` for `chromium`, `puppeteer`, `node`, `npm`, and `mermaid` found:

| Finding | Detail |
|---------|--------|
| `Dockerfile.worker:33` | `apk add nodejs npm chromium` |
| `Dockerfile.worker:36` | `npm install -g @mermaid-js/mermaid-cli` |
| `Dockerfile.worker:39вЂ“40` | `PUPPETEER_SKIP_CHROMIUM_DOWNLOAD` / `PUPPETEER_EXECUTABLE_PATH` env vars |
| `worker/tasks/conversion.py:288вЂ“297` | Calls `alteryx2talend analyze --diagram mermaid --diagram-output {path}.mmd` via `subprocess.run()` |
| `worker/tasks.py:960вЂ“969` | Identical call (duplicate in monolithic tasks file) |
| `worker/tasks/s3.py:231` | Sets `content_type = "text/x-mermaid"` when uploading `.mmd` files to S3 |

**Key finding: Chromium and mermaid-cli are never called by the Python worker.** The worker calls the Go binary to generate a `.mmd` text file, then uploads that file to S3 where the frontend renders it in the browser using mermaid.js. There is no `mmdc` invocation, no headless browser launch, and no puppeteer usage anywhere in the Python codebase. Node.js and npm exist solely to install mermaid-cli, which is itself unused.

**Dockerfile.worker changes (Phase 6):**

```dockerfile
# Before
RUN apk add --no-cache ca-certificates nodejs npm chromium
RUN npm install -g @mermaid-js/mermaid-cli
ENV PUPPETEER_SKIP_CHROMIUM_DOWNLOAD=true \
    PUPPETEER_EXECUTABLE_PATH=/usr/bin/chromium-browser

# After вЂ” remove the above three lines entirely
RUN apk add --no-cache ca-certificates
# Diagram generation handled natively by the Go binary via go-mermaid.
```

**Worker Python upgrade opportunity:** Once the Go binary supports `--diagram-format png`, the worker's `conversion.py` call site should be updated from `--diagram mermaid` (`.mmd` output) to `--diagram-format png` (direct PNG output), eliminating the frontend's dependency on mermaid.js for thumbnail rendering. This is a separate task from Phase 6 but should be planned alongside it.
